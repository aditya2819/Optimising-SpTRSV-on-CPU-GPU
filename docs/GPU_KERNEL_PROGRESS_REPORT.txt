GPU New SpTRSV Kernel Progress Report
Date: 2026-03-31

Goal
Build a clean GPU sparse triangular solve benchmark from scratch, starting with a correctness-first baseline and iterating toward a stronger custom kernel that stays competitive as matrix size grows.

Starting Point
We reset the GPU work into a fresh folder:
- GPU New/sptrsv_kernel.cu
- GPU New/sptrsv_kernel.cuh
- GPU New/sptrsv_kernel.h
- GPU New/sptrsv_benchmark.cu

This new version began with only three paths:
1. CPU reference solver
2. naive GPU kernel
3. cuSPARSE baseline

That gave us a clean baseline for correctness and timing.

Kernel Progression

1. GPU Naive
What it was:
- one GPU thread solves the whole triangular system sequentially

Why we started here:
- simplest correctness baseline
- easiest custom GPU implementation to verify against CPU and cuSPARSE

What we learned:
- correct, but extremely slow
- no meaningful GPU parallelism

2. GPU Coalesced
What changed:
- one CUDA block walks each row with coalesced CSR reads
- shared-memory reduction combines partial sums and the diagonal

Why:
- improve memory behavior
- use many threads for the inner row walk while preserving correctness

What we learned:
- this immediately became the strongest custom kernel
- it stayed the best custom kernel across much of the sweep

3. GPU ThreadRow
What changed:
- one thread handles one row
- host launches rows in order

Why:
- simple row-specialized baseline
- useful as a building block for later adaptive versions

What we learned:
- better than naive
- still weak overall
- too much launch overhead and too little per-row parallelism

4. GPU WarpRow
What changed:
- one warp handles one row
- warp lanes walk the row coalescently
- warp shuffle reduction combines partial sums

Why:
- better fit for medium row sizes
- explicit warp cooperation is natural for CSR row work

What we learned:
- consistently stronger than ThreadRow
- still too launch-heavy because rows were launched one by one

5. GPU BlockRow
What changed:
- one block handles one row
- coalesced row walk plus shared-memory reduction

Why:
- try a heavier row kernel for longer rows
- see whether larger rows benefit from block-wide cooperation

What we learned:
- not useful at small sizes
- becomes competitive once rows get large enough
- still suffers from one-launch-per-row overhead

6. GPU AdaptiveRow
What changed:
- adaptive dispatcher chooses:
  - ThreadRow for tiny rows
  - WarpRow for medium rows
  - BlockRow for larger rows

Why:
- no single row kernel was best for all row sizes
- wanted one balanced row-based custom path

What we learned:
- conceptually better than any single fixed row kernel
- but still underperformed because it launched one kernel per row
- threshold tuning alone could not fix that structural problem

Why AdaptiveRow Was Not Enough
Profiling with Nsight Systems showed the real problem:
- old AdaptiveRow at n=8192: 32780 kernel launches
- old AdaptiveRow at n=32768: 131084 kernel launches

So the issue was not mainly row-kernel choice.
The issue was launch overhead from row-by-row host dispatch.

7. GPU RowBucketed
What changed:
- classify rows into thread / warp / block buckets
- compress contiguous runs of rows with the same bucket into segments
- launch one kernel per segment, not one kernel per row

Why:
- keep the good row-shape selection from AdaptiveRow
- remove the launch storm that profiling exposed

What we learned:
- this was the biggest structural improvement after Coalesced
- it dramatically reduced launch count
- it became a much stronger balanced custom kernel

Profiler-Guided Evidence

Old AdaptiveRow:
- n=8192:
  - 32780 cudaLaunchKernel calls
- n=32768:
  - 131084 cudaLaunchKernel calls

RowBucketed:
- n=16384:
  - 841 cudaLaunchKernel calls
- n=32768:
  - 841 cudaLaunchKernel calls

Interpretation:
- RowBucketed largely solved the original launch-overhead problem
- after this change, the remaining gap to cuSPARSE was no longer mostly launch cost
- the remaining issue became matrix-wide execution efficiency and global scheduling

Overall Conclusions
1. Naive was only a correctness baseline.
2. Coalesced was the first major success and remained the strongest single custom kernel.
3. ThreadRow, WarpRow, and BlockRow were useful stepping stones for understanding row-shape tradeoffs.
4. AdaptiveRow proved that row-size-based selection was a good idea, but also exposed that row-by-row dispatch was fundamentally too expensive.
5. RowBucketed was the profiler-driven improvement that turned adaptive selection into a much more scalable custom path.

Why We Needed Another Step After RowBucketed
Profiling showed that RowBucketed fixed the original launch-storm problem, but it did not fully solve the larger-size scaling problem.

Profiler evidence from Nsight Systems:
- Old AdaptiveRow:
  - n=8192: 32780 kernel launches
  - n=32768: 131084 kernel launches
- RowBucketed:
  - n=16384: 841 kernel launches
  - n=32768: 841 kernel launches

This was a major improvement, but profiling also showed that after launch count dropped, the remaining cost shifted:
- RowBucketed n=16384:
  - cudaLaunchKernel: 7.94 ms
  - cudaDeviceSynchronize: 38.61 ms
- RowBucketed n=32768:
  - cudaLaunchKernel: 8.71 ms
  - cudaDeviceSynchronize: 92.11 ms

Interpretation:
- launch overhead was no longer the dominant issue
- the remaining issue was the amount of time spent waiting for the bucketed kernels to finish
- in other words, we had improved row-shape selection and launch batching, but we still were not exposing enough matrix-wide parallelism

That is why the next step was not more threshold tuning. The next step was to make the execution dependency-aware at the level/frontier scale.

8. GPU LevelAware
What changed:
- compute dependency levels on the host
- within each level, split rows into:
  - thread bucket
  - warp bucket
  - block bucket
- process those row lists in parallel inside each dependency level

Why:
- RowBucketed still processed contiguous row ranges in order
- that preserved correctness and reduced launches, but it left parallelism on the table
- rows in the same dependency level can be solved together safely

What we learned:
- this is the first step toward a true frontier/level-aware adaptive solver
- it directly follows from the profiling result that the remaining bottleneck after bucketing was matrix-wide scheduling, not launch count alone
- the isolated first implementation is correct, but still early and not yet tuned

Why we kept profiling after LevelAware
Once LevelAware was added, family-by-family profiling showed that it was not a universal winner.

What Nsight showed on isolated family cases at n=4096:
- wide-level:
  - LevelAware worked well
  - low enough level count and enough rows per level to amortize overhead
- random:
  - LevelAware was strong, but still had avoidable overhead
  - there were still many launches and repeated host-side orchestration costs
- banded and block:
  - LevelAware was poor
  - too many narrow levels
  - too many per-level allocations and synchronizations

The key lesson was:
- LevelAware was structurally right for wide-frontier and moderately irregular matrices
- but it still carried unnecessary per-level overhead on random matrices

9. GPU RandomAware
What changed:
- built a random-family-specific variant on top of LevelAware
- kept the same dependency-level approach, but reduced overhead in two ways:
  - device row-list buffers are allocated once and reused across levels
  - thread / warp / block row-list launches run on separate CUDA streams within each level
- retuned the bucket thresholds for random structure:
  - thread bucket: nnz <= 2
  - warp bucket: nnz <= 64
  - block bucket: nnz > 64

Why:
- profiling showed that generic LevelAware on random was not losing because the level idea was wrong
- it was losing because there was still too much per-level setup overhead

Profiler evidence that motivated RandomAware
Generic LevelAware on random, n=4096:
- 166 cudaMalloc calls
- 166 cudaFree calls
- 161 cudaLaunchKernel calls

Interpretation:
- too much repeated allocation churn for per-level row lists
- too much host-side orchestration for a family where the level structure itself was already useful

What happened after the change
RandomAware on random, n=4096:
- 7 cudaMalloc calls
- 8 cudaFree calls
- 103 cudaLaunchKernel calls

Measured result on the family sweep:
- random, n=4096:
  - LevelAware: 9.5292 ms
  - RandomAware: 4.8856 ms
  - cuSPARSE: 42.2106 ms

Interpretation:
- RandomAware kept the strength of LevelAware on random
- but removed much of the per-level allocation and launch overhead
- that produced about a 1.95x speedup over the generic LevelAware kernel on that case

What profiling at larger random sizes showed
We then profiled RandomAware against cuSPARSE on the large random case to see where the remaining gap comes from.

Important note:
- isolated Nsight runs in this environment include substantial first-use/runtime overhead
- so the most reliable comparison is the API breakdown, not the raw wall-clock number from the isolated profile

RandomAware on random, n=32768:
- 803 cudaLaunchKernel calls
- 2172 cudaStreamSynchronize calls
- cudaLaunchKernel total API time: 5.54 ms
- cudaStreamSynchronize total API time: 26.08 ms

cuSPARSE on random, n=32768:
- 10 cudaLaunchKernel calls
- 1 cudaDeviceSynchronize call
- cudaLaunchKernel total API time: 37.91 ms
- cudaDeviceSynchronize total API time: 2.42 ms

Interpretation of the large-size gap
- RandomAware no longer loses because of allocation churn
- it now loses because it still has too much fine-grained orchestration:
  - hundreds of launches
  - thousands of stream synchronizations
- cuSPARSE appears to use a much coarser execution schedule with far fewer synchronization points

What performance is still being left on the table
- per-level stream synchronization is still too frequent
- the random-family solver still breaks the work into too many small frontier steps
- row-list buffering helped, but the solver still does not batch dependency frontiers as aggressively as cuSPARSE

What this suggests for the next random-family optimization
- fuse consecutive small levels for random matrices
- reduce the number of stream synchronizations
- move from per-level submission toward coarser frontier chunks
- keep the row-list reuse, because that part already fixed one real bottleneck

What happened when we tried the first two random-specific optimizations
We explicitly tried:
1. fusing consecutive small levels
2. reducing host-side synchronization by replacing per-level stream synchronizations with event-based ordering and a final stream sync

What happened:
- the first fusion attempt was not safe
- it broke correctness because later-level work inside a fused chunk could observe incomplete results from earlier levels in another bucket stream
- the isolated correctness check showed this immediately through a large residual

What we kept:
- we reverted the unsafe level-fusion part
- we kept the safer event-based ordering and final synchronization structure

Next family target
After wide-level and random, the next family to optimize is power-law.

Why power-law is next:
- LevelAware was already one of the strongest custom kernels there
- the family still has meaningful parallel structure
- unlike banded or chain, it is still a good GPU target
- unlike block, it is less dominated by narrow-frontier overhead pathologies

So the progression of family-specific attention is now:
1. wide-level:

11. GPU BlockAware
What changed:
- added a block-family-specific path based on BCSR rather than CSR
- instead of making the kernel choose a block size dynamically, the host analyzes the matrix first
- candidate block sizes:
  - 2x2
  - 4x4
  - 8x8
- the host picks the block size using block occupancy analysis, then converts CSR to BCSR once and launches a BCSR solve kernel

Why:
- the block family is essentially patchy clusters of nonzeros
- that makes it the most natural candidate for a block-compressed storage format
- the goal was to reduce scalar CSR metadata overhead and exploit local dense structure inside patches

How the analysis works:
- for each candidate block size, we simulate block packing
- we compute:
  - number of nonempty blocks
  - number of real entries inside those blocks
  - total stored block entries
  - occupancy = real entries / stored block entries
- we choose the block size with the best occupancy, with a slight bias toward larger blocks when occupancies are close

What we learned from the first BlockAware version
- the first version only considered 2x2 and 4x4
- it was correct, but still clearly behind RowBucketed
- so we expanded the analysis to include 8x8 as well, since the synthetic block family uses much larger block patches

Representative results after adding 2x2 / 4x4 / 8x8 analysis
block, n=4096:
- RowBucketed: 9.8501 ms
- BlockAware: 59.7357 ms
- cuSPARSE: 32.8781 ms

block, n=16384:
- RowBucketed: 52.7033 ms
- BlockAware: 508.3194 ms
- cuSPARSE: 57.3172 ms

Interpretation
- BlockAware is correct, but the current kernel is still not competitive
- at 4096 it improved over the first BCSR attempt, but it still trails RowBucketed badly
- at 16384 it becomes much worse than both RowBucketed and cuSPARSE

Why the current BlockAware version still loses
- the format analysis and block-size selection idea are reasonable
- the main weakness is the current execution model:
  - one cooperative CUDA block processes the BCSR solve sequentially across block rows
- that means we reduce CSR metadata overhead, but we still leave too much matrix-wide parallelism unused

Conclusion for BlockAware
- host-side block-size analysis is the right policy mechanism for the block family
- but the current BCSR execution kernel is too sequential to capitalize on it
- so the next improvement for block would not be changing the selector again
- it would be writing a stronger block-parallel BCSR solve kernel

What we changed next for PowerLawAware
We simplified the first PowerLawAware design to cut orchestration overhead:
- removed the separate thread bucket entirely
- changed to a warp/block-only split:
  - warp bucket: nnz <= 24
  - block bucket: nnz > 24
- collapsed the execution onto a single work stream per level
- removed the event/wait choreography between multiple streams

Why:
- profiling said the main remaining issue was no longer the row-kernel choice itself
- it was the amount of event/wait and submission traffic around heavy-tail rows
- so the next step was to trade some overlap for a much simpler submission model

  - PowerLawAware: 2.2411 ms
  - cuSPARSE: 1.8095 ms
- power-law, n=32768:
  - PowerLawAware: 3.1485 ms

What we learned from this revision
- the simplification helped at every tested size
- PowerLawAware remains the best custom kernel for power-law
- but cuSPARSE still keeps a meaningful lead at the larger sizes
- this means the profile-guided change was directionally right, but not enough to fully close the gap

Checking whether added machinery is still hurting performance
We profiled the simplified PowerLawAware and compared it to the earlier event-heavy style.

What improved:
- the extra event/wait machinery was hurting performance
- after simplification:
  - cudaEventRecord traffic disappeared from the hot path
  - cudaStreamWaitEvent traffic disappeared from the hot path
- this confirms that the earlier multi-stream/event choreography was overhead, not value, for the power-law family

What is still hurting now:
- the new dominant overhead is per-level stream synchronization
- current simplified PowerLawAware at n=32768:
  - 724 cudaStreamSynchronize calls
  - 775 cudaLaunchKernel calls
  - 775 cudaMemcpyAsync calls

Interpretation:
- we successfully removed one layer of harmful top-level machinery
- but we still have another expensive orchestration layer:
  - one stream synchronization per active dependency level
  - one async row-list upload per active level bucket

So the answer is:
- yes, some of what we added on top was hindering performance
- specifically the old event/wait choreography
- after removing that, the next remaining hindrance is per-level stream synchronization and per-level row-list submission

Safe overhead-removal sweep
We then removed only the overhead we believed could be removed without changing dependency correctness:
- LevelAware:
  - removed repeated per-level cudaMalloc/cudaFree
  - switched to reusable row-list buffers
- RandomAware:
  - simplified from multi-stream/event orchestration to a single work stream per level

What the sweep showed
- The effect was mixed.
- LevelAware generally benefited from buffer reuse.
- RandomAware generally regressed after losing the more aggressive stream structure.

- in practice:

- not all top-level machinery is bad
- some of it was overhead
- some of it was actually buying useful overlap
- so future cleanup should be selective rather than blanket simplification

15. FamilyAware final adaptive kernel
After working family by family, we built one top-level family-aware adaptive path.

Goal:
- inspect matrix structure on the host
- classify the matrix into one of the synthetic family shapes we studied
- dispatch directly to the best family-specific kernel

Final mapping used by the selector:
- random -> RandomAware
- banded -> BandedAware
- chain -> RowBucketed
- wide-level -> RandomAware
- block -> RowBucketed
- power-law -> PowerLawAware

What changed during classifier tuning
The first classifier version was too expensive and too inaccurate:
- it reused heavy analysis such as level computation and multi-size block analysis
- that made the FamilyAware timings much worse than the chosen kernel timings
- it also misclassified some block and power-law cases

We then simplified the classifier:
- use cheap row statistics:
  - average row nnz
  - coefficient of variation of row nnz
  - max row nnz
  - bandwidth
- add one cheap row-contiguity feature:
  - adjacentFrac, the fraction of consecutive off-diagonal entries within rows

Why adjacentFrac helped:
- random had very low adjacentFrac
- block had moderate adjacentFrac because of clustered patches
- wide-level had very high adjacentFrac, but very low row-nnz variance
- this gave a cheap way to separate random, block, and wide-level without expensive block simulation

Timing fix

We also fixed the negative-time glitch seen in some long real-matrix runs:
- root cause: `timeMs(...)` was using `std::chrono::high_resolution_clock`
- on this platform that clock is not guaranteed to be monotonic
- so host-side elapsed time could become nonsensical if the clock source jumped

Fix:
- switched benchmark timing to `std::chrono::steady_clock`

Why this is the right fix:
- `steady_clock` is monotonic
- it is the correct host clock for elapsed-time measurement
- this removes bogus negative wall-clock timings without changing solver behavior
